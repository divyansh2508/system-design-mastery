System Design · Module 7 — Messaging Systems

Message Queue & Consumer Groups

Watch a Kafka-style broker in motion. A producer hashes each message by key into one of N partitions; a consumer group of M consumers each owns a subset of partitions and pulls messages, advancing its offset. Change the partition and consumer counts, flood the producer, or make one consumer slow — and see exactly where lag piles up. The lessons: partitions are the unit of parallelism, extra consumers beyond the partition count sit idle, and a slow consumer only backs up its own partitions.

◆ Run these four experiments

1

Isolate the lag. Start balanced (4 partitions, 4 consumers) and flip Slow consumer on. Only C0's partition reddens and backs up — every other partition stays green. Lag is per-partition; one slow consumer can't stall the group.

2

Parallelism is capped by partitions. Drop to 1 partition and push the produce rate up. Add consumers — they just sit idle and lag keeps climbing. One partition is read by exactly one consumer, so nothing parallelizes.

3

Now unlock it. From that jammed state, add partitions one at a time. Each rebalance hands a fresh partition to an idle consumer, they start pulling in parallel, and the backlog drains. Partitions = the scaling knob.

4

Too many consumers. Set 3 partitions and 6 consumers — 3 consumers own nothing and read zero. You can't parallelize past the partition count; adding consumers beyond it is wasted capacity, not more throughput.
