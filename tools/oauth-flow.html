◆ Run these experiments

1

Follow the channels. Step to 8 and watch the colors flip at step 5. Everything sensitive — the verifier and the tokens — moves on the back channel. Only the authorization code ever rides the browser (front channel). No token ever appears in a URL.

2

Steal the code. Turn on Attacker steals the code, keep PKCE on, and step to 6. The exchange is rejected — the attacker has the code but not the verifier. That's PKCE stopping a real interception attack.

3

Remove the guardrail. Leave the attacker on and switch PKCE off. Replay to step 6: the same stolen code is now redeemed and the attacker walks away with tokens. This is the pre-PKCE world the spec was written to fix.

4

Check the math. Note that code_challenge = SHA256(verifier) — but you can't run SHA-256 backwards. Hit Reset a few times and watch fresh, real values regenerate. The server stores only the challenge; only the holder of the verifier can finish.
