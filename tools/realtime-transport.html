System Design · Module 10 — Real-Time APIs

Real-Time Transport Compare

Three ways to get server events into a browser, racing side by side. A server emits events at a rate you control; watch WebSocket push instantly over a two-way channel, SSE stream one-way over a held connection, and long-polling re-open a fresh request for every single event. Push the rate up and see long-poll's reconnect gap turn into backlog and latency — the exact tradeoff you defend in the interview.

Where do webhooks fit? They don't — a webhook is the server-to-server cousin: one backend HTTP-POSTs an event to another backend's URL. It can't push to a browser tab, so it isn't in this race. These three are the browser-facing options.

What's happening

All three transports deliver in about the same time at a low event rate — the difference isn't speed, it's cost. Turn up the rate to expose it.

◆ Run these experiments

1

Watch the cost, not the clock. At 5 events/s all three latencies read ~50 ms — long-poll is not slower here. But watch the "requests opened" line: long-poll racks up one request per event while WebSocket and SSE stay at 1. The tax is overhead, not latency.

2

Break long-poll. Drag the rate past ~10 events/s — its throughput ceiling. Now events arrive faster than the client can re-request, the server buffer backs up, and long-poll's latency climbs into the hundreds of ms, then seconds. WebSocket and SSE never flinch.

3

Prove bidirectionality. Hit Client → server. Only WebSocket carries it back up the same open channel. SSE and long-poll flash a red "separate request" — their stream is server→client only, so upstream needs a whole extra HTTP call.

4

Pick the right tool. Need two-way (chat, multiplayer, presence)? WebSocket. One-way feed (notifications, live scores, token streaming)? SSE — simpler, auto-reconnects, rides plain HTTP. Neither available through a hostile proxy? Long-poll, the universal fallback you pay overhead for.
