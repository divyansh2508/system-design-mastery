        What's happening

        ▸ Show the math (payload, byte-by-byte)

        How the bytes are counted: JSON string field = "key":"value", → key + value + 6 B (quotes, colon, comma). Protobuf field = tag (1 B for field #1–15) + length (1 B) + value bytes — the field name never travels, only its number. That's the whole trick: protobuf drops keys, quotes, and braces, and packs small ints as varints. Long string values (URLs, timestamps, bios) cost the same bytes in both — which is why binary is ~1.5–2× smaller on rich objects, not 10×.

        ◆ Run these experiments

        1

        Watch round-trips explode. Switch to User + 3 orders: REST jumps to 4 round-trips (the N+1 trap) while GraphQL and gRPC stay at 1. Now drag RTT to 200 ms — REST's time-to-data hits ~800 ms. On slow links, round-trips hurt more than bytes.

        2

        Bust the "10× smaller" myth. In Fetch a user profile, gRPC is only ~1.9× smaller than REST — not 10×. Long string values cost the same bytes in binary; protobuf only strips the field names and structural chars. Read the "×N vs gRPC" line under each payload.

        3

        GraphQL isn't always smaller. Still in Fetch a user profile, notice GraphQL is the largest payload — its {data:…} envelope adds bytes when you request every field. GraphQL only wins when you trim fields — flip to the Mobile scenario and watch it collapse.

        4

        Feel the over-fetch. In Mobile, REST ships all 11 fields when the screen uses 2 (~77% wasted). GraphQL and gRPC send only what's asked. Compare the payload bars — then ask why REST is still the pick for a public, cacheable API.
