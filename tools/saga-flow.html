What's happening

Press ▷ Run saga to push one distributed transaction through four services. Each performs a local commit with no global lock — so every forward step must define an inverse (its compensation) for the saga to undo it if a later step fails.

Coordination model — Orchestration

A central orchestrator tells each service what to do next and issues the compensations on failure. Easy to monitor and reason about — but the orchestrator holds all the workflow logic and can become a coordination bottleneck.

◆ Run these experiments

1

Watch a happy path. Leave Fail at None and run. All four commit left-to-right and go green — the saga ends COMMITTED with 0 distributed locks held. That's the whole promise: cross-service consistency without 2PC.

2

Fail at Shipping. Order, Payment, and Inventory commit — then Shipping fails. Watch three compensations run in reverse: Restore Inventory → Refund Payment → Cancel Order. Note the failed step is not compensated — its own local transaction already rolled back.

3

Fail at Payment vs Inventory. Fail early (Payment) and only Cancel Order runs; fail later (Inventory) and two compensations run. The further the saga gets, the more there is to unwind — so put risky, hard-to-undo steps last.

4

Toggle Orchestration vs Choreography and re-run the same failure. The steps and compensations are identical — but read the log: an orchestrator commanding each service versus services reacting to each other's events. Same saga, different control flow.
