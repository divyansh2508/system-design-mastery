Consistent Hashing Ring

How do you spread keys across servers so that adding or removing a server barely disturbs anything? Watch the ring: each key belongs to the first node clockwise. Add a node, remove a node, and compare how many keys move here vs. under naive hash % N — the difference is the whole reason this technique exists.

◆ Run these experiments

1

Add a node. Watch "moved · consistent" stay small (~1/N of keys) while "naive hash % N" jumps to almost everything. That churn is a cache-wipe in production.

2

Remove a node. Its keys go to the next node clockwise — every other key stays put. Naive %N reshuffles nearly all of them again.

3

Drop virtual nodes to 1. Load goes lumpy — one node owns a huge arc. Crank it to 40 and watch the load even out. That's what vnodes are for.

4

Connect it back. This is the fix for the Load Balancer's "IP Hash trap" — sticky routing without the remap-everything problem when a server dies.
