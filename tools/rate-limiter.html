Rate Limiter Algorithms

Four ways to answer one question: "is this request allowed right now?" Feed a stream of requests into a limiter, fire bursts at it, and watch each algorithm decide. See Token Bucket permit bursts, Leaky Bucket smooth output to a flat rate, Fixed Window leak 2× at the boundary, and Sliding Window close that hole. Every request is either admitted (green) or throttled with a 429 (red).

What's happening

Token Bucket starts full. It refills at the limit rate and hands out one token per request — a burst can drain it instantly, then throttling kicks in until it refills.

◆ Run these experiments

1

Bursts pass, then don't. On Token Bucket, keep incoming below the limit (admit rate ~100%). Hit Send burst: the whole burst is admitted at once — up to the bucket size — then requests throttle until the bucket refills at the limit. That's the burst allowance token bucket is famous for.

2

Spiky in, smooth out. Switch to Leaky Bucket and push incoming above the limit. Input is jagged, but watch the output drip leave at a constant rate — the queue fills, overflow gets 429'd, and downstream sees a perfectly flat stream. This is traffic shaping.

3

The boundary leak. On Fixed Window, watch the countdown to the reset line. Send burst just before it hits 0, then again just after. Read "Max admitted · rolling 1s" — it jumps to ~2× your limit. The per-window counter reset let a double dose slip through the seam.

4

The fix. Switch to Sliding Window and repeat the same two bursts. No matter how you time them, "Max admitted · rolling 1s" never exceeds the limit — the rolling log checks the true last second, so the boundary hole is gone.
