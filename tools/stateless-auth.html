Module 2 · Stateless architecture

The guest list vs. the signed ticket

Every request has to answer one question: who are you? There are two ways to answer it — and the difference only becomes obvious once it happens a few hundred times. So make it happen. Click things.

The guest-list server keeps no memory of you between requests, so it re-looks-you-up every time — twice on a role-protected route (once for “who are you”, once for “what may you do”). That’s the 2 measured in the AksharShruti code. The ticket carries the answers, so the server just checks the seal — 0 lookups.

02 Why can’t you just fake the ticket?

A signed ticket (a JWT) is three parts joined by dots: header.payload.signature. The payload literally states who you are — so why not change "user" to "superuser"? Try it.

"sub""user_42"
"username""ravi"
"role"
"exp"in 15 min
🔒Seal valid — server trusts this ticket.

Change the role and the seal breaks — the signature no longer matches the contents, and the server throws the ticket out. Only the server (which holds the secret key) can make a valid seal, so you can’t promote yourself. That’s what makes it safe to trust the ticket without asking the database.

03 The catch: nothing is free

Trusting the ticket means the server stops checking the database — so a change made after the ticket was issued doesn’t apply until a new ticket is issued. Promote Ravi and watch the two worlds disagree.

🚫 admin panel: denied
Signed ticketreads the ticket
ticket says: user
🚫 admin panel: denied

Both say “user”. Now promote him.

The guest list updates instantly; the ticket stays stale until Ravi gets a fresh one. The fix isn’t to go back to lookups — it’s to keep tickets short-lived (yours last 15 minutes), so stale info self-corrects fast. That’s the trade: speed & scale, in exchange for changes that apply on a short delay.

04 Why any of this matters

The real payoff. You want more traffic capacity, so you add servers behind a load balancer. Flip between the two worlds.
