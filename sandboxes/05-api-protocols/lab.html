Sandbox Lab·Module 9 — APIs

REST · GraphQL · gRPC side by side

Spin up one tiny Node service that answers the same user query three ways — then fetch it over each protocol and watch the payload shapes diverge with your own eyes.

Prerequisites

Have Docker Desktop running (whale icon steady, not spinning). Then open a terminal and cd into this folder — every command below is run from sandboxes/05-api-protocols/.

01  What you'll build

A single container — node:20-alpine — running one process that exposes the identical user record through three API styles at once: REST and GraphQL on port 3000, and gRPC on port 50051.

One data store, three doors. Because the data is held constant, every difference you see in the response is caused purely by the protocol — that's what makes this a fair comparison instead of three unrelated demos.

                      ┌──────────────────────────────────────────┐
  curl ──REST─────▶   │  api  (node:20-alpine container)         │
                      │                                          │
  curl ──GraphQL──▶   │   ┌────────────────┐                     │
                      │   │  Express app   │  :3000  HTTP/1.1    │
  node ──gRPC─────▶   │   │  REST + GraphQL│                     │
                      │   └───────┬────────┘                     │
                      │           │ reads                        │
                      │   ┌───────▼────────┐                     │
                      │   │  users{} store │ ◀── single source   │
                      │   └───────▲────────┘     of truth        │
                      │           │ reads                        │
                      │   ┌───────┴────────┐                     │
                      │   │ grpc-js server │  :50051  HTTP/2     │
                      │   └────────────────┘                     │
                      └──────────────────────────────────────────┘

The one idea

REST gives you a fixed shape per URL. GraphQL lets the client pick the fields. gRPC trades human-readable JSON for a typed, compact binary contract over HTTP/2. Same data — three very different envelopes.

02  The stack

The whole lab is one service. Here is the docker-compose.yml that runs it — modern Compose schema, no obsolete version: key, image tag pinned, and one comment per service:

# API Protocols sandbox — one Node.js service, three ways to fetch the same 'user'.
# Modern Compose schema (no top-level `version:` key — it is obsolete).

services:
  # Single Node.js service (built from ./Dockerfile, node:20-alpine).
  # Serves the SAME user resource over REST + GraphQL (HTTP/:3000) and gRPC (HTTP/2/:50051).
  api:
    build: .
    image: api-protocols-lab:1.0
    container_name: api-protocols
    ports:
      - "3000:3000"    # REST (GET /users/:id) + GraphQL (POST /graphql) over HTTP/1.1
      - "50051:50051"  # gRPC (UserService/GetUser) over HTTP/2
    restart: unless-stopped

Alongside it in this folder sit the files the image builds from — you don't edit them, but it's worth knowing what each is for:

FileRole
DockerfileBuilds the node:20-alpine image and installs deps
package.jsonDeps: express, graphql, @grpc/grpc-js, @grpc/proto-loader
server.jsThe one process: REST + GraphQL on :3000, gRPC on :50051
user.protoThe gRPC contract — a typed User message + GetUser RPC
client.jsA tiny bundled gRPC client (so you don't need grpcurl)

03  Bring it up

Build the image and start the container in the background:

docker compose up -d --build

The first run downloads node:20-alpine and installs the npm deps, so give it a minute. When it returns, confirm the service is healthy and both listeners are live:

docker compose ps
docker compose logs api

You want to see the container Up with ports 3000 and 50051 published, and these two log lines:

REST + GraphQL listening on http://0.0.0.0:3000
gRPC listening on 0.0.0.0:50051

Port already in use?

If curl localhost:3000 returns HTML from some other app, you already have a dev server on port 3000. Stop it first, or change the left side of the port mappings (e.g. "3001:3000") in docker-compose.yml and re-run up -d. The gRPC exercise uses 50051 and is unaffected.

04  The exercises

Fetch user 1 — Ada Lovelace — three ways. Run them in order and keep the outputs on screen so you can compare the envelopes at the end.

A · RESTThe whole object, every time

A plain GET against a resource URL. The server decides the shape; you get all of it:

curl -s localhost:3000/users/1

Response — the entire record, all six fields, whether you needed them or not:

{"id":"1","name":"Ada Lovelace","email":"ada@analytical.engine","age":36,"city":"London","role":"Mathematician"}

What to observeYou asked for a user and got everything the server models about a user. If your screen only needs the name, you still paid to serialize and download age, city, and role. That is over-fetching — the classic REST tax.

B · GraphQLYou choose the fields — no over-fetch

One POST to a single /graphql endpoint. The query names exactly the fields you want — here just name and email:

curl -s -X POST localhost:3000/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{ user(id: \"1\") { name email } }"}'

Response — precisely the two fields requested, nested under data, and nothing else:

{"data":{"user":{"name":"Ada Lovelace","email":"ada@analytical.engine"}}}

What to observeSame data store, same user — but age, city, and role never left the server. Add a field to the query (try { user(id: \"1\") { name email city } }) and watch the response grow to match. The client, not the server, now controls the payload shape.

C · gRPCA typed binary contract over HTTP/2

There's no easy raw curl for gRPC — the wire format is binary protobuf over HTTP/2, and the fields aren't even named on the wire. So call it with the tiny client bundled in the image, straight from inside the container:

docker compose exec api node client.js

Response — decoded from the compact binary frame back into an object for you to read:

{
  "id": "1",
  "name": "Ada Lovelace",
  "email": "ada@analytical.engine",
  "age": 36,
  "city": "London",
  "role": "Mathematician"
}

What to observeThe payload looks like the REST one, but how it travelled is the whole point. On the wire, field names were replaced by the numbered slots in user.proto (name = 2, email = 3…), so the bytes are far smaller and strictly typed. Break the contract on purpose — docker compose exec api node client.js 99 — and you get a typed NOT_FOUND status, not a stray 404 page.

Put the three envelopes next to each other

This is the payoff — the same user, three shapes:

ProtocolWho picks the fieldsWire formatWhat came back
RESTServer (fixed per URL)JSON / HTTP 1.1All 6 fields — likely over-fetch
GraphQLClient (per query)JSON / HTTP 1.1Exactly the 2 fields asked for
gRPCContract (the .proto)Binary protobuf / HTTP 2Typed message, compact bytes

That single table is the entire trade-off space in miniature: REST is simplest and cache-friendly but rigid; GraphQL kills over- and under-fetching at the cost of a query planner; gRPC is fastest and strictest but needs generated stubs and isn't browser-native. You just watched all three, on identical data.

05  Teardown

Stop and remove the container, its network, and the named volumes in one shot:

docker compose down -v

Run docker compose ps afterward — it should list nothing for this project. The built image lingers in your local cache; drop it too with docker image rm api-protocols-lab:1.0 if you want a spotless machine.

06  What this taught + where next

You now have felt, not just read, the core protocol trade-off:

  • REST — resource URLs return a server-defined shape. Dead simple and cacheable, but you over-fetch when the client needs less.
  • GraphQL — one endpoint, client-selected fields. It eliminates over- and under-fetching, at the price of server-side query execution.
  • gRPC — a .proto contract compiled to typed stubs, riding compact binary over HTTP/2. Fastest and strictest; ideal service-to-service, awkward straight from a browser.

In a system design interview, this is exactly the muscle you flex when asked "which API style for this edge, and why?" — mobile client on a slow network leans GraphQL, an internal high-throughput service mesh leans gRPC, a public third-party API usually leans REST.
