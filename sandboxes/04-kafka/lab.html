Sandbox Lab·Module 7 — Messaging

Kafka Partitions & Consumer Groups

Run a real single-broker Kafka in KRaft mode on your own machine, split a topic into three partitions, then watch a consumer group rebalance and its lag move — the exact mechanics behind every "add a queue to absorb load" answer.

Prerequisites

Docker Desktop must be running (check the whale icon / docker info succeeds). Then open a terminal and cd into this folder — every command below is run from here:

cd "System Design/sandboxes/04-kafka"

01  What you'll build

One Kafka broker, one topic named orders with three partitions, and one consumer group g1 that you'll grow from one member to two — so you can see partitions get reassigned live.

Kafka 3.x runs in KRaft mode: the cluster's metadata quorum lives inside Kafka itself, so there is no ZooKeeper to run. Our single node wears both hats — it is the broker (serves produce/consume) and the controller (owns the metadata log). That is the whole stack: one container.

A partition is the unit of parallelism and ordering. A topic's messages are spread across its partitions; order is guaranteed within a partition, not across them. A consumer group is a set of consumers that share the work: Kafka assigns each partition to exactly one member of the group. So with 3 partitions, a group can have up to 3 active consumers pulling in parallel — a 4th would sit idle. Add or remove a member and Kafka rebalances the assignment.

The whole point

You never write a line of application code here. You drive Kafka's own bundled shell scripts (under /opt/kafka/bin) through docker compose exec — the same tools operators use in production to poke a real cluster.

02  The stack

The entire lab is one file — docker-compose.yml. It's a single service using the official Apache image, pinned to 3.7.0, configured for single-node KRaft. Read the comments; every environment variable earns its place.

# Sandbox 04 — Kafka Partitions & Consumer Groups
# Single-broker Apache Kafka in KRaft mode (NO ZooKeeper).

services:
  # The one node plays BOTH roles (broker + controller) — quorum self-hosted.
  broker:
    image: apache/kafka:3.7.0          # official Apache image, pinned tag
    container_name: kafka-sandbox
    hostname: broker
    ports:
      - "9092:9092"                    # bootstrap port, published to the host
    environment:
      # --- node identity + roles (KRaft) ---
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_CONTROLLER_QUORUM_VOTERS: 1@localhost:9093
      # --- listeners: bind on 0.0.0.0, advertise localhost ---
      KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT
      KAFKA_INTER_BROKER_LISTENER_NAME: PLAINTEXT
      # --- single-node replication factors (must be 1 with one broker) ---
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
      KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR: 1
      KAFKA_TRANSACTION_STATE_LOG_MIN_ISR: 1
      # --- lab ergonomics: rebalance instantly when a 2nd consumer joins ---
      KAFKA_GROUP_INITIAL_REBALANCE_DELAY_MS: 0
      KAFKA_NUM_PARTITIONS: 1
    # Broker is "up" once the admin API answers a topic list.
    healthcheck:
      test: ["CMD-SHELL", "/opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 --list || exit 1"]
      interval: 10s
      timeout: 10s
      retries: 6
      start_period: 20s

Why advertise localhost?

A client first asks the broker "where do I send data?" and the broker replies with its advertised address. Because we drive everything from inside the container with docker compose exec, localhost:9092 is the address that always resolves — for both the in-container CLI and a host client hitting the published port.

03  Bring it up

Start the broker in the background:

docker compose up -d

The first run pulls the image (~350 MB) and formats the KRaft storage automatically — give it a moment. Confirm the container is healthy:

docker compose ps

You want to see kafka-sandbox with state running and, after ~20 s, healthy. Tail the logs if you're curious:

docker compose logs -f broker   # Ctrl-C to stop tailing (the broker keeps running)

Look for Kafka Server started. A quick end-to-end sanity check — list topics (there are none yet, so this returns nothing and exits 0):

docker compose exec broker /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server localhost:9092 --list

04  The exercises

Do these in order. A couple of steps run long-lived, foreground processes (a producer, then consumers) — those need their own terminal windows, all cd'd into this same folder. You'll end up with up to three terminals open at once.

4.1 Create the topic — 3 partitions terminal 1

docker compose exec broker /opt/kafka/bin/kafka-topics.sh \
  --create \
  --topic orders \
  --partitions 3 \
  --replication-factor 1 \
  --bootstrap-server localhost:9092

Then describe it to see the partition layout:

docker compose exec broker /opt/kafka/bin/kafka-topics.sh \
  --describe --topic orders --bootstrap-server localhost:9092

What to observe

Three lines, one per partition (Partition: 0, 1, 2). Each shows Leader: 1 and Replicas: 1 — our single broker leads every partition. ReplicationFactor: 1 means no copies (fine for a lab, never for production). This is the parallelism you just carved out.

4.2 Produce a first batch terminal 1

Open the console producer. It reads one message per line from your keyboard until you exit:

docker compose exec broker /opt/kafka/bin/kafka-console-producer.sh \
  --topic orders --bootstrap-server localhost:9092

At the > prompt, type a few lines, pressing Enter after each:

> order-1
> order-2
> order-3
> order-4
> order-5
> order-6

Press Ctrl-C (or Ctrl-D) to quit the producer and return to your shell.

What to observe

No key was given, so Kafka's default partitioner spreads these messages across partitions 0/1/2 in batches. You've now seeded the topic — the messages sit durably in their partitions, waiting for a consumer.

Optional: route by key

Want deterministic placement? Produce keyed records — the same key always lands on the same partition: --property parse.key=true --property key.separator=:, then type lines like eu:order-7 and us:order-8.

4.3 Consume in group g1 terminal 2

In a second terminal (also cd'd here), start a consumer that joins group g1 and reads the topic from the very beginning:

docker compose exec broker /opt/kafka/bin/kafka-console-consumer.sh \
  --topic orders \
  --group g1 \
  --from-beginning \
  --bootstrap-server localhost:9092

What to observe

All six messages print (order may look shuffled — that's expected, because ordering is per-partition, not global). Leave this consumer running. Right now one member owns all three partitions, so it receives everything.

4.4 Add a second consumer — watch the rebalance terminal 3

In a third terminal, start a second consumer in the same group g1 (no --from-beginning this time — it should pick up new work, not replay):

docker compose exec broker /opt/kafka/bin/kafka-console-consumer.sh \
  --topic orders \
  --group g1 \
  --bootstrap-server localhost:9092

Now go back to terminal 1 and produce a fresh batch so there's new traffic to split:

docker compose exec broker /opt/kafka/bin/kafka-console-producer.sh \
  --topic orders --bootstrap-server localhost:9092
> order-7
> order-8
> order-9
> order-10
> order-11
> order-12

What to observe

The instant the second consumer joined, Kafka rebalanced: the three partitions are now split across the two members — one gets two partitions, the other gets one. The new messages arrive split between the two terminals instead of all in one. You just watched a consumer group scale out horizontally, live. (Kill one consumer with Ctrl-C and the survivor picks up all three partitions again — another rebalance.)

4.5 Inspect offsets & LAG terminal 1

This is the operator's-eye view. While both consumers are still running, describe the group:

docker compose exec broker /opt/kafka/bin/kafka-consumer-groups.sh \
  --describe --group g1 --bootstrap-server localhost:9092

You'll see one row per partition. To make LAG jump above zero, stop both consumers (Ctrl-C in terminals 2 and 3), produce another batch, then describe again:

# terminal 1 — with NO consumers running, add a backlog
docker compose exec broker /opt/kafka/bin/kafka-console-producer.sh \
  --topic orders --bootstrap-server localhost:9092
# type order-13 … order-18, then Ctrl-C

docker compose exec broker /opt/kafka/bin/kafka-consumer-groups.sh \
  --describe --group g1 --bootstrap-server localhost:9092

What to observe

Read the columns per partition: CURRENT-OFFSET (how far the group has read), LOG-END-OFFSET (how far the log has been written), and LAG = LOG-END-OFFSET − CURRENT-OFFSET — the backlog waiting to be processed. With the consumers stopped and 6 new messages produced, total LAG across the partitions is 6. Restart a consumer in g1 and it drains the backlog; describe once more and LAG falls back to 0. That single number is what on-call engineers page on.

05  Teardown

Ctrl-C any consumers/producers still in the foreground, then stop and remove everything — container, network, and volumes — with one command:

docker compose down -v

The -v flag removes the broker's storage too, so the topic, its partitions, and all committed offsets are gone. Next docker compose up -d gives you a pristine cluster.

06  What this taught

You now have hands-on, not-just-slideware intuition for the load-bearing ideas of Module 7:

  • Partitions are parallelism + ordering. Splitting orders into three let a group process it three-ways; order held within each partition, never across.
  • Consumer groups scale out — up to the partition count. Two members shared three partitions; a hypothetical fourth would idle. Partition count is the ceiling on consumer parallelism.
  • Rebalancing is automatic and observable. Membership changed, assignments moved — no code, no downtime.
  • LAG is the health metric. LOG-END-OFFSET − CURRENT-OFFSET tells you whether consumers are keeping up. Rising lag is the universal "we're falling behind" signal.
  • KRaft removes ZooKeeper. A modern single-binary Kafka — one image, one process, both roles.
