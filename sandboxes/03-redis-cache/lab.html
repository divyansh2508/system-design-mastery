Sandbox Lab · Module 5-6 — Caching

Redis Caching & Eviction

Run a real Redis node capped at a tiny 10 MB, then watch — with your own eyes, live from redis-cli — a key expire on its TTL and old keys get evicted by LRU when the cache fills up.

Prerequisites

Docker Desktop must be running (check the whale icon in your menu bar / system tray). Then open a terminal and cd into this lab's folder — every command below runs from there:

cd "System Design/sandboxes/03-redis-cache"

01  What you'll build

A single Redis server — the same in-memory key/value store that fronts real production read paths — deliberately shrunk to 10 MB so its two most important cache behaviours become visible in seconds.

Caches don't grow forever. Two mechanisms keep them bounded, and this lab makes you feel both: TTL expiry (a key you tag with a lifetime vanishes when the clock runs out) and eviction (when the memory cap is hit, Redis throws out keys to make room). We set --maxmemory-policy allkeys-lru, so the key it throws out is the Least Recently Used one — exactly the policy a read cache wants.

            docker compose exec redis redis-cli
                            │
                            ▼
        ┌───────────────────────────────────┐
        │   redis   (image: redis:7-alpine)  │   host :6379  ->  container :6379
        │   service "redis"                  │
        │                                    │
        │   --maxmemory 10mb                 │   capped tiny ON PURPOSE, so the
        │   --maxmemory-policy allkeys-lru   │   limit is easy to hit and eviction
        │                                    │   is visible within seconds
        └───────────────────────────────────┘
      in-memory key/value store  ·  TTL expiry  ·  LRU eviction

There is no application code in this lab — you are the application. You talk to Redis directly with redis-cli, running each command inside the container via docker compose exec. That keeps every mechanism in plain sight instead of hidden behind a client library.

02  The stack

One file defines the whole lab. docker-compose.yml — a single official Redis node, started with the two flags that make it a bounded, LRU cache:

# Redis caching & eviction sandbox — System Design, Module 5-6.
# Bring up : docker compose up -d
# Tear down: docker compose down -v
# No top-level `version:` key — Compose v2 treats this as the modern schema.

services:
  # One Redis node, capped tiny (10 MB) with LRU eviction so TTL expiry and
  # allkeys-lru eviction are visible live via `docker compose exec redis redis-cli`.
  redis:
    image: redis:7-alpine
    container_name: sandbox-redis
    command: ["redis-server", "--maxmemory", "10mb", "--maxmemory-policy", "allkeys-lru"]
    ports:
      - "6379:6379"          # host:container — reach it from your machine too
    restart: unless-stopped

Why this is the real thing

This is not a toy shim — it's the same official redis-server that runs in production, driven by the exact flags (maxmemory + a maxmemory-policy) an SRE tunes on a real cache tier. Everything you observe here — TTLs, eviction counts, INFO stats — behaves identically at scale.

03  Bring it up

Pull the image and start Redis in the background:

docker compose up -d

Verify it's healthy — you should see one service, running, mapping 0.0.0.0:6379->6379/tcp:

docker compose ps

Now confirm Redis answers and that our two flags actually took effect:

docker compose exec redis redis-cli PING
# -> PONG

docker compose exec redis redis-cli CONFIG GET maxmemory
# -> 1) "maxmemory"  2) "10485760"   (10 MB, in bytes)

docker compose exec redis redis-cli CONFIG GET maxmemory-policy
# -> 1) "maxmemory-policy"  2) "allkeys-lru"

What to observe

PONG means the server is up. The two CONFIG GET lines are the proof that matters: the cap is 10485760 bytes (10 MB) and the policy is allkeys-lru. Those two settings are what make Exercises 2 and 3 do something interesting.

04  The exercises

Four steps. Run the commands exactly, and read the “what to observe” note after each — that's where the learning is.

Exercise 1 — SET and GET a key

The two commands the whole cache is built on

A cache is, at heart, a dictionary. You put a value under a key, then you fetch it back. Store one, then read it:

docker compose exec redis redis-cli SET user:42 "Ada Lovelace"
# -> OK

docker compose exec redis redis-cli GET user:42
# -> "Ada Lovelace"

What to observe

SET replies OK and GET hands the value straight back from RAM — no disk, no query planner. This is the ~100 ns memory read that makes a cache worth having. The key name (user:42) is just a string; the namespace:id convention is pure convention, and it's the one every real codebase uses.

Exercise 2 — Give a key a TTL and watch it expire

SET … EX, then watch TTL count down to nothing

Cached data goes stale, so you attach a time-to-live. SET … EX 15 stores a key that self-destructs in 15 seconds — perfect for a session token or a short-lived cache entry. Store one, then sample its TTL three times, five seconds apart:

docker compose exec redis redis-cli SET session:abc "token-xyz" EX 15
# -> OK

# Sample the remaining life 3 times, 5s apart, and watch it fall:
for i in 1 2 3; do docker compose exec redis redis-cli TTL session:abc; sleep 5; done
# -> (integer) 15   (roughly — could be 14)
# -> (integer) 10
# -> (integer) 5

Now let the last few seconds elapse, then try to read the key that has expired:

docker compose exec redis redis-cli GET session:abc
# -> (nil)          the key is gone — expired and deleted

docker compose exec redis redis-cli TTL session:abc
# -> (integer) -2   -2 means "no such key"

What to observe

The TTL counts down in real time, and the instant it hits zero the key is deleted — GET returns (nil), not a stale value. Note the two special TTL return codes you'll see everywhere: -2 = the key doesn't exist; -1 = the key exists but has no expiry set (try TTL user:42 from Exercise 1 to see -1). This automatic expiry is how caches stay fresh without anyone sweeping them by hand.

Exercise 3 — Overflow the 10 MB cap and watch LRU eviction

Fill Redis past its limit, then read the eviction stats

First, take a baseline of an empty-ish cache so you have something to compare against:

docker compose exec redis redis-cli DBSIZE
# -> (integer) 1   (just user:42 from Exercise 1 — small)

docker compose exec redis redis-cli INFO stats | grep evicted_keys
# -> evicted_keys:0   nothing evicted yet

Now the fill loop. We build one ~128 KB value with a pure-shell doubling trick (no extra tools needed), then SET 500 big keys — roughly 64 MB of data aimed at a 10 MB cache. It's a plain for loop calling redis-cli SET, run inside the container in a single exec (500 separate host-side execs would crawl):

docker compose exec redis sh -c '
  # Build a ~128 KB value: start with "x" and double it 17 times (2^17 = 131072 bytes).
  val=x; for i in $(seq 1 17); do val=$val$val; done

  # Write 500 big keys. Far more than 10 MB, so Redis MUST start evicting.
  for i in $(seq 1 500); do redis-cli SET "bigkey:$i" "$val" >/dev/null; done

  echo "wrote 500 keys x ~128KB into a 10MB cap"
'
# -> wrote 500 keys x ~128KB into a 10MB cap   (takes a few seconds)

Now read what actually happened — how many keys survived, how many got evicted, and how memory stayed pinned under the cap:

docker compose exec redis redis-cli DBSIZE
# -> (integer) ~70   FAR fewer than the 500+1 you wrote — the rest were evicted

docker compose exec redis redis-cli INFO stats | grep evicted_keys
# -> evicted_keys:430+   Redis threw out this many keys to stay under 10 MB

docker compose exec redis redis-cli INFO memory | grep -E 'used_memory:|used_memory_human:|maxmemory:|maxmemory_human:|maxmemory_policy:'
# -> used_memory:~10000000        parked just below the cap
# -> used_memory_human:~9.5M
# -> maxmemory:10485760
# -> maxmemory_human:10.00M
# -> maxmemory_policy:allkeys-lru

Finally, prove it was the Least Recently Used keys that got dropped. The very first keys you wrote are the coldest; the last ones are the warmest:

docker compose exec redis redis-cli EXISTS bigkey:1
# -> (integer) 0   the oldest, coldest key is gone — evicted first

docker compose exec redis redis-cli EXISTS bigkey:500
# -> (integer) 1   the newest key is still here

What to observe

You wrote ~64 MB but used_memory settled just under 10 MB — Redis refuses to exceed its cap. evicted_keys jumped from 0 into the hundreds, and DBSIZE is a fraction of what you inserted: every write past the limit forced an eviction. And bigkey:1 (coldest) is gone while bigkey:500 (warmest) survives — that's allkeys-lru doing its job. Exact numbers vary run to run: LRU is approximate (Redis samples a handful of keys rather than tracking a perfect global order), and memory includes per-key overhead — so treat DBSIZE and evicted_keys as “roughly,” not to the digit.

Exercise 4 — The cache-aside read pattern (the reason this all matters)

How a real app actually uses everything above

You've now seen the three primitives a cache is made of: fast GET/SET, TTL freshness, and bounded memory via eviction. Cache-aside (also called lazy loading) is the pattern that wires them into an application's read path. It's the most common caching strategy in the industry, and it works like this:

  1. A request comes in for some data. The app checks the cache first — GET user:42.
  2. Cache hit: Redis returns the value instantly (~100 ns from RAM). The app returns it and is done. The database was never touched.
  3. Cache miss ((nil)): the data isn't cached. The app reads the source of truth — the database — which is slower (a network hop plus a disk/query).
  4. Before returning, the app populates the cache with what it just fetched — SET user:42 <value> EX 300 — usually with a TTL so the entry can't go stale forever.
  5. The next read of user:42 is now a hit, served from Redis until its TTL expires or LRU evicts it under memory pressure.
# The read path, in pseudocode — every line maps to a redis-cli command you just ran:
value = redis.GET(key)                 # check cache first
if value is nil:                       # MISS
    value = db.query(key)              # fall back to the source of truth
    redis.SET(key, value, EX=300)      # populate cache (with a TTL) for next time
return value                           # HIT path returns here directly

The payoff

Cache-aside is why a read-heavy service can serve millions of requests without melting its database: the hot working set lives in Redis (the ~100 ns path), and only misses — cold or first-time reads — reach the DB. TTL keeps entries from going stale; LRU eviction keeps the cache pinned to the hottest data when memory is tight. You just operated all three by hand — that's exactly the machinery running under a real app's cache layer.

05  Teardown

Stop and remove the container, the network, and any named volumes in one command:

docker compose down -v

What to observe

Everything this lab created is gone — docker compose ps shows nothing for this project, and port 6379 is free again. Because Redis kept all your keys in RAM (no persistence configured), tearing down wipes the data too — which is exactly how a pure cache should behave. The one source file (docker-compose.yml) stays on disk, so docker compose up -d rebuilds the whole thing in seconds.

06  What this taught — and the simulator pairing

In a handful of commands you drove the full mechanics of a production cache tier:

  • GET / SET — the in-memory key/value core; the ~100 ns read that makes caching worth it.
  • TTL expiry — SET … EX gives entries a lifetime so cached data can't go stale forever; -2/-1 are the “gone” / “no expiry” signals.
  • Bounded memory + LRU eviction — maxmemory plus allkeys-lru keep the cache pinned to its hottest keys and drop the coldest, visible in INFO stats (evicted_keys) and INFO memory (used_memory).
  • Cache-aside — the read pattern (check cache → miss → read DB → populate cache) that ties all of the above into a real application.
