Sandbox Lab · Module 2 — Scalability

Nginx Load Balancer

Put a real Nginx reverse proxy in front of three identical backends, then curl it and watch requests round-robin — and watch Nginx route around a backend you kill, live, on your own machine.

Prerequisites

Docker Desktop must be running (check the whale icon in your menu bar / system tray). Then open a terminal and cd into this lab's folder — everything below runs from there:

cd "System Design/sandboxes/01-nginx-load-balancer"

01  What you'll build

One load balancer (Nginx) fronting three interchangeable backends — the exact shape every scalable web tier takes.

The backends are traefik/whoami, a tiny official image whose whole job is to echo the container's own hostname on every request. That's the trick that makes load balancing visible: each reply literally tells you which of the three servers answered, so you can watch the rotation with your own eyes instead of trusting a diagram.

                         curl localhost:8080
                                  │
                                  ▼
                        ┌───────────────────┐
                        │       nginx        │   host :8080  ->  container :80
                        │     service "lb"   │
                        └─────────┬─────────┘
                                  │   upstream "backends"  (round-robin)
                  ┌───────────────┼───────────────┐
                  ▼               ▼               ▼
           ┌────────────┐  ┌────────────┐  ┌────────────┐
           │  whoami1   │  │  whoami2   │  │  whoami3   │
           │    :80     │  │    :80     │  │    :80     │
           └────────────┘  └────────────┘  └────────────┘
              traefik/whoami — each echoes its own hostname

All four containers share a private Docker bridge network, so Nginx reaches the backends by their service names (whoami1, whoami2, whoami3) via Docker's built-in DNS. Only the load balancer's port is published to your host — the backends are never exposed directly, exactly as in production.

02  The stack

Two files define the whole lab. First, docker-compose.yml — four services on one network, only the LB published:

# Nginx reverse-proxy load balancer in front of three whoami backends.
# Bring up : docker compose up -d
# Tear down: docker compose down -v

services:
  # Backend 1 — traefik/whoami echoes the container hostname on every request.
  whoami1:
    image: traefik/whoami:v1.10.2
    hostname: whoami1
    networks: [labnet]

  # Backend 2 — identical; you will `docker compose stop whoami2` mid-lab.
  whoami2:
    image: traefik/whoami:v1.10.2
    hostname: whoami2
    networks: [labnet]

  # Backend 3 — identical.
  whoami3:
    image: traefik/whoami:v1.10.2
    hostname: whoami3
    networks: [labnet]

  # Load balancer — official nginx, round-robins the three backends.
  # nginx.conf is mounted read-only; container :80 -> host :8080.
  lb:
    image: nginx:1.27-alpine
    ports:
      - "8080:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on: [whoami1, whoami2, whoami3]
    networks: [labnet]

# Private bridge network so nginx resolves whoami1/2/3 by name.
networks:
  labnet:
    driver: bridge

Second, nginx.conf — it defines an upstream pool of the three backends and proxies every request to it. With no method specified, Nginx uses round-robin:

events {}

http {
    # The pool. No method line -> round-robin: 1 -> 2 -> 3 -> 1 ...
    upstream backends {
        server whoami1:80;
        server whoami2:80;
        server whoami3:80;
    }

    server {
        listen 80;
        location / {
            proxy_pass http://backends;
            proxy_set_header Host            $host;
            proxy_set_header X-Real-IP       $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            # If a backend refuses/errors, retry the next one in the pool.
            proxy_next_upstream error timeout http_502 http_503 http_504;
        }
    }
}

Why this is the real thing

This is not a toy shim — it's the same nginx.conf shape that fronts production web tiers. The upstream { } block is the load-balancing pool, and proxy_pass is the reverse proxy. Everything you learn here transfers directly.

03  Bring it up

Pull the images and start all four containers in the background:

docker compose up -d

Verify the stack is healthy — you should see four services, all running, with the LB mapping 0.0.0.0:8080->80/tcp:

docker compose ps

Now confirm the front door answers:

curl -s localhost:8080 | grep Hostname
# -> Hostname: whoami1   (or whoami2 / whoami3 — whichever is next in rotation)

What to observe

The whoami response body includes a Hostname: line that names the container that served the request. That single line is your window into the load balancer for the rest of this lab.

04  The exercises

Three short experiments. Run the commands exactly, and read the “what to observe” note after each — that's where the learning is.

Exercise 1 — Watch round-robin rotate

Fire six requests in a row and watch the server change

Because there's no session affinity and no weighting, Nginx hands each successive request to the next backend in the pool, then wraps around:

for i in $(seq 1 6); do curl -s localhost:8080 | grep Hostname; done

What to observe

The hostnames cycle in a clean, repeating order — whoami1 → whoami2 → whoami3 → whoami1 → whoami2 → whoami3. That perfectly even 1-2-3-1-2-3 pattern is round-robin. Run it again; the rotation simply continues from wherever it left off.

Exercise 2 — Kill a backend, watch Nginx route around it

Stop whoami2, then curl again

This is the whole point of a load balancer: if one server dies, traffic keeps flowing to the survivors. Stop the middle backend, then send another six requests:

docker compose stop whoami2

for i in $(seq 1 6); do curl -s localhost:8080 | grep Hostname; done

What to observe

You still get six successful 200 responses — but whoami2 never appears; only whoami1 and whoami3 answer. On the first request that lands on the stopped backend, Nginx gets a connection refused, and thanks to proxy_next_upstream it silently retries the next server, so the client never sees an error. Nginx then marks whoami2 as failed and, by default (max_fails=1, fail_timeout=10s), stops sending it traffic for the next 10 seconds. So don't expect to see errors — the correct outcome is that failover is invisible to the caller.

Bring it back

Start the backend again and it rejoins the rotation:

docker compose start whoami2

for i in $(seq 1 6); do curl -s localhost:8080 | grep Hostname; done
# whoami2 is back in the 1-2-3 rotation (give it a second after fail_timeout elapses)

What to observe

Once fail_timeout lapses, Nginx tentatively sends a request back to whoami2; it succeeds, the backend is un-quarantined, and all three names appear again. You just watched a node leave and rejoin a pool with zero client-facing downtime.

Exercise 3 — Tune the passive health check

Add max_fails / fail_timeout, then reload Nginx

What you saw in Exercise 2 was Nginx's passive health checking: it never actively probes the backends — it just watches real requests and quarantines a server that fails. The defaults are max_fails=1, fail_timeout=10s. You tune them right on the server lines of the upstream. Open nginx.conf and change the pool to:

upstream backends {
    # Tolerate 3 failures inside a 15s window before quarantining a backend,
    # and keep it out of rotation for 15s once tripped.
    server whoami1:80 max_fails=3 fail_timeout=15s;
    server whoami2:80 max_fails=3 fail_timeout=15s;
    server whoami3:80 max_fails=3 fail_timeout=15s;
}

Then reload Nginx without restarting the container — the config is picked up in place:

# Validate the edited config first, then hot-reload it
docker compose exec lb nginx -t
docker compose exec lb nginx -s reload

What the two knobs mean

max_fails is how many failed attempts (within fail_timeout) it takes to consider a backend down; fail_timeout is both that counting window and how long the backend stays quarantined afterward. Raise max_fails to tolerate transient blips without ejecting a server; lower it to fail fast. Re-run Exercise 2 (stop whoami2) to feel the difference — with max_fails=3, Nginx tolerates more errors before pulling a backend out.

DirectiveDefaultWhat it controls
max_fails1Failed attempts within the window before the backend is marked down
fail_timeout10sThe counting window, and how long the backend stays quarantined

05  Teardown

Stop and remove the containers, the network, and the named volumes in one command:

docker compose down -v

What to observe

Everything this lab created is gone — docker compose ps now shows nothing for this project, and port 8080 is free again. The two source files (docker-compose.yml, nginx.conf) stay on disk, so docker compose up -d rebuilds the whole thing in seconds whenever you want it back.

06  What this taught — and the simulator pairing

In a few commands you exercised the core of a scalable web tier:

  • Reverse proxy + upstream pool — one public entry point (proxy_pass) fanning out to many private backends.
  • Round-robin — the default, session-less way to spread load evenly across identical servers.
  • Passive failover — Nginx routes around a dead backend automatically, and proxy_next_upstream hides the failure from the client.
  • Health-check tuning — max_fails / fail_timeout trade fail-fast responsiveness against tolerance for transient blips.
