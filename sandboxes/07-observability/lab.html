Sandbox Lab·Module 14 — Observability

Prometheus + Grafana

Stand up the real Prometheus + Grafana metrics stack, scrape a live machine with node-exporter, then turn raw counters into a query, a dashboard panel, and a firing alert — the exact loop behind every SLI, SLO, and error budget.

Prerequisites

Docker Desktop must be running. Then open a terminal and cd into this folder (sandboxes/07-observability/) — every command below is run from here, where the docker-compose.yml and prometheus.yml live. You'll also open two URLs in your browser: localhost:9090 (Prometheus) and localhost:3000 (Grafana).

01  What you'll build

Three official containers wired into the canonical metrics pipeline: node-exporter publishes machine metrics, Prometheus scrapes and stores them every 15 seconds, and Grafana sits on top for dashboards and alerting. No app code — you are the operator, driving it from the terminal and two web UIs.

This is the shape of every production monitoring setup you'll ever meet. The pull model is the key idea: Prometheus reaches out and scrapes each target's /metrics endpoint on a schedule — targets don't push. Grafana never touches the targets; it only ever queries Prometheus.

   ┌───────────────┐   query    ┌────────────────────┐
   │   Grafana     │  (PromQL)  │     Prometheus     │
   │   :3000       │ ─────────▶ │     :9090          │
   │  dashboards + │            │  stores series,    │
   │  alert rules  │            │  scrapes every 15s │
   └───────────────┘            └─────────┬──────────┘
                                         │ pull GET /metrics
                          ┌──────────────┴──────────────┐
                          ▼                             ▼
                ┌────────────────────┐      ┌────────────────────┐
                │   node-exporter    │      │  prometheus itself │
                │   :9100/metrics    │      │  :9090/metrics     │
                │  CPU · mem · disk  │      │  (its own health)  │
                └────────────────────┘      └────────────────────┘

By the end you'll have queried live metrics in Prometheus, connected Grafana to it, drawn a time-series panel of CPU load, and built an alert that actually fires when a target goes down — then tied all of it back to the language of reliability: SLIs, SLOs, and error budgets.

02  The stack

Three services, all official images, zero custom code. Prometheus mounts the prometheus.yml next to it, which tells it what to scrape. Here's the complete docker-compose.yml sitting next to this page:

# Observability sandbox — System Design, Module 14
# The classic metrics stack, all official images, no custom code:
#   node-exporter  → exposes host/kernel metrics at :9100
#   prometheus     → scrapes those metrics every 15s and stores + queries them
#   grafana        → dashboards + alerting on top of Prometheus
# You talk to Prometheus at http://localhost:9090 and Grafana at http://localhost:3000.

services:
  # Prometheus: pulls metrics from itself and node-exporter, stores the time series,
  # and answers PromQL queries. Config is mounted read-only from ./prometheus.yml.
  prometheus:
    image: prom/prometheus:v2.53.0
    container_name: sandbox-prometheus
    ports:
      - "9090:9090"                                        # host:container — open http://localhost:9090
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro # our scrape config
    restart: unless-stopped

  # node-exporter: the canonical exporter for machine metrics (CPU, memory, disk, net).
  # Its cpu/meminfo collectors read the shared-kernel /proc, so metrics like
  # node_cpu_seconds_total work out-of-the-box with no host mounts — keeps this
  # cross-platform on Docker Desktop. (On a bare Linux host you can add
  # `--path.rootfs=/host` + a `/:/host:ro,rslave` mount for full host fidelity.)
  node-exporter:
    image: prom/node-exporter:v1.8.1
    container_name: sandbox-node-exporter
    ports:
      - "9100:9100"                                        # host:container — raw metrics at http://localhost:9100/metrics
    restart: unless-stopped

  # Grafana: connect it to Prometheus as a data source, then build panels + alerts.
  # Login is admin / admin (set explicitly so the box is predictable).
  grafana:
    image: grafana/grafana:11.1.0
    container_name: sandbox-grafana
    ports:
      - "3000:3000"                                        # host:container — open http://localhost:3000
    environment:
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=admin
    depends_on:
      - prometheus
    restart: unless-stopped

And the prometheus.yml it mounts — the whole point is the two scrape jobs and the 15 second interval:

# Prometheus scrape configuration — System Design, Module 14 (Observability)
# Prometheus PULLS metrics from each target on an interval, over HTTP.
# Two jobs below: Prometheus scrapes itself, and it scrapes node-exporter.

global:
  scrape_interval: 15s      # how often to pull /metrics from every target
  evaluation_interval: 15s  # how often to evaluate recording/alerting rules

scrape_configs:
  # 1) Prometheus scraping its own /metrics (localhost = inside the prometheus container).
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]

  # 2) Prometheus scraping node-exporter over the compose network.
  #    "node-exporter" is the service name → Docker DNS resolves it to the container.
  - job_name: "node"
    static_configs:
      - targets: ["node-exporter:9100"]

Why node-exporter:9100 and not localhost

Inside the compose network every service is reachable by its service name as a hostname. From the Prometheus container, node-exporter:9100 resolves to the exporter; localhost:9090 means Prometheus itself. This is the same reason Grafana will point at http://prometheus:9090 later — not localhost.

03  Bring it up

Start all three containers in the background, then confirm each one is answering:

# pull images + start everything (detached)
docker compose up -d

# all three should be "running" / healthy
docker compose ps

# is Prometheus healthy? → "Prometheus Server is Healthy."
curl -s localhost:9090/-/healthy

# is node-exporter producing metrics? → a stream of node_* lines
curl -s localhost:9100/metrics | grep -m1 node_cpu_seconds_total

# is Grafana up? → {"commit":"...","database":"ok","version":"11.1.0"}
curl -s localhost:3000/api/health

What to observe: docker compose ps lists sandbox-prometheus, sandbox-node-exporter, and sandbox-grafana all up. The Prometheus health check returns a healthy line, node-exporter prints a node_cpu_seconds_total sample, and Grafana's health JSON says "database":"ok". First run pulls the images, so give it a minute.

04  The exercises

Five exercises, in order. The first two live in the Prometheus UI (localhost:9090); the last three build the Grafana half (localhost:3000) and end with an alert you make fire on purpose. Keep your terminal handy — the finale is driven from it.

Exercise 1

Confirm the scrape — and meet up, your first SLI

Open http://localhost:9090 in your browser. Go to Status → Targets. You should see both scrape jobs — prometheus (1/1 up) and node (1/1 up) — each with State = UP. Now switch to the Graph tab, type the query below into the expression bar, and press Execute (keep the Table view):

up

What to observe: two rows come back — up{instance="localhost:9090",job="prometheus"} = 1 and up{instance="node-exporter:9100",job="node"} = 1. up is a metric Prometheus synthesises for every target on every scrape: 1 = the scrape succeeded, 0 = it failed. That single number is the simplest possible SLI (Service Level Indicator): is the thing reachable? Remember it — Exercise 5 turns it into an alert.

Exercise 2

Query real machine metrics with PromQL

node-exporter exposes hundreds of series. The most important pattern in all of PromQL is rate() over a counter — a number that only ever goes up (like total CPU seconds). rate() converts it into a per-second change. Still on the Graph tab at localhost:9090, run this and switch to the Graph view:

rate(node_cpu_seconds_total[1m])

You'll get one line per CPU core per mode (user, system, idle, iowait…). That's noisy. Now collapse it into the one number an operator actually watches — percent of CPU that is busy — by measuring idle and subtracting from 100:

100 - (avg(rate(node_cpu_seconds_total{mode="idle"}[1m])) * 100)

What to observe: the first query is a thicket of per-core lines; the second is a single, readable CPU-busy percentage that hovers low when idle and jumps when you do work on your machine. This second expression is a genuine saturation SLI — you just wrote the exact query a real dashboard panel would run. Copy it; you'll paste it into Grafana next.

Exercise 3

Wire Grafana to Prometheus

Open http://localhost:3000 and log in with admin / admin. (If Grafana prompts you to set a new password, click Skip — this is a throwaway lab.) Then add the data source:

  • Left menu → Connections → Data sources → Add data source.
  • Choose Prometheus.
  • In Connection → Prometheus server URL, enter exactly:
http://prometheus:9090
  • Leave everything else default. Scroll down and click Save & test.

What to observe: a green banner — "Successfully queried the Prometheus API." / "Data source is working." If instead you get a connection error, you almost certainly typed localhost — from inside the Grafana container localhost is Grafana, not Prometheus. On the compose network the address is the service name, prometheus.

Exercise 4

Build a time-series panel

Now visualise the SLI from Exercise 2. Create a dashboard and a panel:

  • Left menu → Dashboards → New → New dashboard → Add visualization.
  • Pick your Prometheus data source. The visualization type (top-right) defaults to Time series — leave it.
  • In the query editor, toggle to Code mode and paste the CPU-busy expression:
100 - (avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
  • On the right panel options: set Title to CPU busy %, and under Standard options → Unit choose Percent (0–100).
  • Click Run queries, then Apply (top-right), then the Save dashboard disk icon — name it Node health.

What to observe: a live line chart of CPU utilisation, labelled as a percentage, refreshing as Prometheus scrapes. You've now made an SLI visible — this is the whole job of a dashboard: turn a raw counter into a signal a human can glance at. Open a busy program on your machine and watch the line climb.

Exercise 5

Create an alert — and burn an error budget on purpose

A dashboard is passive; an alert is the thing that pages you. We'll alert on the availability SLI from Exercise 1 — up for the node target — because it maps perfectly onto an SLO. First, the reliability vocabulary this exercise makes concrete:

TermIn this lab
SLIup{job="node"} — is the target being scraped successfully? (1 or 0)
SLOa target you pick, e.g. 99.9% availability over 30 days
Error budget100% − 99.9% = 0.1% ≈ ~43 min/month of allowed downtime
The alertfires the moment up drops to 0 — i.e. you've started spending that budget

Build the rule in Grafana. Left menu → Alerting → Alert rules → New alert rule:

  • Name: node-exporter down (availability SLO).
  • Query (section 1): data source Prometheus, in Code mode set query A to:
up{job="node"}
  • Condition (section 2): keep the two default expressions Grafana adds — B = Reduce (function Last, input A) and C = Threshold (input B). Set the threshold to IS BELOW 1, and make sure C is marked as the alert condition.
  • Evaluation behavior (section): click New folder → sandbox; New evaluation group → name node, interval 10s. Set Pending period to None so it fires immediately, not after a delay.
  • Click Save rule and exit. On Alerting → Alert rules the rule shows state Normal (green) — because the target is up.

Now break it on purpose from your terminal. Stopping node-exporter leaves its target still listed in prometheus.yml, so Prometheus records up=0 for it — exactly a real outage:

# kill the target — Prometheus keeps scraping and now records up=0
docker compose stop node-exporter

# (optional) prove it from the Prometheus side — this returns up ... 0
sleep 20 && curl -s 'localhost:9090/api/v1/query?query=up%7Bjob%3D%22node%22%7D'

Watch Alerting → Alert rules in Grafana. Within ~10–30 seconds the rule flips Normal → Pending → Firing (red). Then bring the target back:

# restore the target — up returns to 1, the alert resolves
docker compose start node-exporter

What to observe: the rule turns red and Firing the moment availability drops to 0, then returns to Normal once the target is back. That firing window is your error budget being spent in real time — every second the target is down eats into the ~43 min/month that a 99.9% SLO allows. A production on-call alert is exactly this: a condition on an SLI that trips before the SLO is breached, so a human can act while there's still budget left. You just built the whole feedback loop end to end.

05  Teardown

When you're done, stop and remove all three containers (and any volumes) so nothing lingers or holds ports 9090/3000/9100:

docker compose down -v

What to observe: Compose stops and removes sandbox-prometheus, sandbox-node-exporter, and sandbox-grafana. The -v drops volumes too, so your Grafana data source, dashboard, and alert rule are wiped — the next docker compose up -d starts from a clean slate.

06  What this taught you

In three containers and a handful of commands you built and drove a complete monitoring pipeline — and, more importantly, connected each moving part to the reliability concept it serves:

You didYou sawThe concept
Status → Targets, query upup = 1Scrape health = an availability SLI
rate(node_cpu_seconds_total[1m])counter → per-second ratePromQL: rates over counters
Add Prometheus data source"Data source is working"Grafana queries Prometheus

The through-line is the reliability loop: instrument (node-exporter) → collect (Prometheus) → visualise (Grafana panel) → alert (rule on an SLI) → act before the error budget runs out. Every observability stack you'll ever run is a variation on what you just built.
