Sandbox Lab·Module 15 — Security

OAuth 2.0 with Keycloak

Spin up a real identity provider, mint your own access token from the OIDC token endpoint with curl, then crack the JWT open and read the exact claims — iss, sub, exp, scope — that the simulator only drew for you.

Prerequisites

Docker Desktop must be running. Open a terminal and cd into this folder before you run anything:

cd "System Design/sandboxes/08-oauth-keycloak"

01  What you'll build

One container: Keycloak, an open-source, production-grade OAuth 2.0 Authorization Server and OpenID Connect Provider. No database to wire up, no app code — you configure everything through its admin console, then talk to the standard OIDC endpoints yourself.

The goal is to make the abstract concrete. In the OAuth Flow Visualizer you watched a token get issued and saw its claims in a panel. Here you'll actually request one from a live token endpoint and decode the JWT with your own hands — the same iss, sub, exp, and scope fields, now coming off a real server on your machine.

02  The stack

A deliberately tiny docker-compose.yml — a single, official Keycloak image in dev mode. Read the comments; every line earns its place.

# OAuth 2.0 / OpenID Connect sandbox — a single Keycloak identity provider.
# `start-dev` gives you an in-memory H2 DB, plain HTTP, and relaxed hostname
# checks — perfect for a throwaway local lab. No custom code anywhere.
#   Bring up:  docker compose up -d      Tear down: docker compose down -v

services:
  keycloak:
    image: quay.io/keycloak/keycloak:latest   # official Keycloak (Quarkus) image
    container_name: sd-keycloak
    command: ["start-dev"]
    environment:
      # First-boot admin for the master realm / admin console. Keycloak 26+
      # renamed these to KC_BOOTSTRAP_ADMIN_*; we set BOTH pairs so the lab
      # works whether `latest` is a pre-26 or a 26+ release.
      - KEYCLOAK_ADMIN=admin
      - KEYCLOAK_ADMIN_PASSWORD=admin
      - KC_BOOTSTRAP_ADMIN_USERNAME=admin
      - KC_BOOTSTRAP_ADMIN_PASSWORD=admin
    ports:
      - "8080:8080"   # admin console + OIDC endpoints, published to the host

Why no resource server?

We deliberately skip a protected API. The whole point of this lab is the token — how it's issued and what's inside it. Adding a resource server buys reliability problems, not learning. Keycloak alone gives you a real Authorization Server, which is the piece that actually matters here.

03  Bring it up

Start the container in the background, then watch it boot:

docker compose up -d
docker compose logs -f keycloak
What to observeWait for a line like Keycloak 26.x … started in 12.3s. Listening on: http://0.0.0.0:8080. First boot pulls the image and can take a minute. Press Ctrl-C to stop following the logs (the container keeps running).

Confirm the OIDC endpoints are live before touching the console — every realm publishes a discovery document, and the built-in master realm is up immediately:

curl -s http://localhost:8080/realms/master/.well-known/openid-configuration \
  | python3 -m json.tool | head -n 12
What to observeA JSON blob whose issuer is http://localhost:8080/realms/master and which lists a token_endpoint, authorization_endpoint, and jwks_uri. Note there is no /auth in these paths — the Quarkus-based Keycloak dropped that legacy prefix.

04  The exercises

Steps 1–3 happen in the admin console (point-and-click); steps 4–5 happen in your terminal. Do them in order.

Exercise 1 — Log in and create the demo realm

Open http://localhost:8080/admin and sign in with admin / admin. A realm is an isolated tenant — its own users, clients, and signing keys. Never build in master; it's for administering Keycloak itself.

1
Open the realm switcher

Top-left dropdown (it currently reads master) → Create realm.

2
Name it demo

Type demo in Realm name, leave Enabled on, click Create.

What to observeThe switcher now reads demo and the URL is /admin/master/console/#/demo. Everything you do from here lives inside this realm. Its issuer will be http://localhost:8080/realms/demo — remember that; you'll see it in the token.

Exercise 2 — Create a confidential client with Direct Access Grants

A client is an application allowed to request tokens. "Confidential" means it authenticates with a secret; "Direct Access Grants" enables the password grant we'll use to fetch a token straight from curl.

1
Clients → Create client

Client type OpenID Connect, Client ID = demo-client. Next.

2
Capability config

Turn Client authentication ON (this makes it confidential). Ensure Direct access grants is checked. Next → Save.

3
Grab the secret

Open the Credentials tab → copy the Client secret. You'll paste it in Exercise 4.

What to observeA Credentials tab only appears because you flipped on Client authentication — a public client (the kind the simulator uses) wouldn't have one. That secret is how a backend proves which app is asking, on top of proving which user.

Exercise 3 — Create a test user

You need a human to authenticate. Create alice and give her a permanent password.

1
Users → Add user

Username = alice. Click Create. (Email/name optional.)

2
Credentials → Set password

Set password alice (twice). Turn Temporary OFF, then Save.

The #1 gotcha

If Temporary is left On, the very next step fails with Account is not fully set up — Keycloak insists the user reset the password interactively first, which curl can't do. Temporary Off is what makes the password grant work.

Exercise 4 — Request an access token from the token endpoint

Now the OAuth part. Paste your client secret, then hit the demo realm's token endpoint with the Resource Owner Password grant. This is the OAuth message the simulator labels POST /token:

# 1) Paste the Client secret you copied in Exercise 2:
export CLIENT_SECRET='PASTE_YOUR_CLIENT_SECRET_HERE'

# 2) Ask Keycloak for a token, pretty-print the whole JSON response:
curl -s -X POST \
  http://localhost:8080/realms/demo/protocol/openid-connect/token \
  -d 'grant_type=password' \
  -d 'client_id=demo-client' \
  -d "client_secret=$CLIENT_SECRET" \
  -d 'username=alice' \
  -d 'password=alice' \
  -d 'scope=openid profile email' \
  | python3 -m json.tool
What to observeA JSON object with access_token (a long xxxxx.yyyyy.zzzzz string — that's the JWT), expires_in: 300 (5 minutes), a refresh_token, token_type: "Bearer", and scope: "openid profile email". If you get {"error":"invalid_grant"}, re-check Exercise 3 (Temporary Off); invalid_client means the secret is wrong.

Now capture just the access token into a variable so you can decode it next:

export TOKEN=$(curl -s -X POST \
  http://localhost:8080/realms/demo/protocol/openid-connect/token \
  -d 'grant_type=password' \
  -d 'client_id=demo-client' \
  -d "client_secret=$CLIENT_SECRET" \
  -d 'username=alice' \
  -d 'password=alice' \
  -d 'scope=openid profile email' \
  | python3 -c 'import sys,json; print(json.load(sys.stdin)["access_token"])')

echo "$TOKEN"
What to observeThree base64url chunks joined by dots: header.payload.signature. It only looks encrypted — a JWT is signed, not encrypted. Anyone can read it; only Keycloak's private key can forge one. You're about to prove that.

Exercise 5 — Decode the JWT and read its claims

The middle segment is the payload — the claims. Split on the dots, take chunk 2, and base64url-decode it. (JWT uses URL-safe base64 without padding, so the one-liner restores the padding before decoding.)

echo "$TOKEN" | cut -d. -f2 | python3 -c '
import sys, base64, json
seg = sys.stdin.read().strip()
seg += "=" * (-len(seg) % 4)          # restore base64url padding
claims = json.loads(base64.urlsafe_b64decode(seg))
print(json.dumps(claims, indent=2))'
What to observeThe decoded claims — the same fields the simulator's token panel showed, now real:
ClaimWhat you'll seeMeans
isshttp://localhost:8080/realms/demoWho issued it — your realm
suba UUIDThe subject — alice's stable user id
expa Unix timestampExpiry — ~300s after iat
scopeopenid profile emailWhat the token is allowed to do
azpdemo-clientAuthorized party — which client asked
preferred_usernamealiceHuman-readable identity

Or paste it into a decoder

Run echo "$TOKEN", copy the string, and paste it into jwt.io — the same claims render in a live panel, header and payload split out. (It decodes in-browser, but never paste a production token into a website.)

Bonus — verify the exp is real

Wait five minutes, then re-run the decode of the same old $TOKEN against a protected call, or just compare exp to date +%s. Then fetch a fresh one with the Exercise 4 command — a new exp, a new signature. That short lifetime is why refresh tokens exist.

Prefer the public-client path?

If you'd rather skip the secret: in Exercise 2 leave Client authentication OFF (a public client, like the simulator's). Then in Exercise 4 simply delete the client_secret line — everything else is identical.

05  Teardown

One command removes the container and its ephemeral state. Because start-dev keeps everything in memory, your realm, client, and user vanish — a clean slate next time you bring it up.

docker compose down -v
What to observeRemoving sd-keycloak … done. Re-running docker compose ps shows nothing. The -v flag drops any volumes so no state survives.

06  What this taught you

You stood up a real OAuth 2.0 Authorization Server, registered a client, created a user, and pulled a signed access token off the standard OIDC token endpoint — then decoded it and read every claim. The four abstractions on the simulator's canvas (authorization server, client, token endpoint, JWT) are now things you've operated by hand.

How it pairs with the simulator

Deliberate contrast — and it's the whole point. This lab used the Resource Owner Password grant: the user's password travels through the client to get a token. That's the legacy shortcut, and it's exactly what the 🔐 OAuth Flow Visualizer exists to eliminate. The simulator walks the Authorization Code flow with PKCE, where the password never reaches the app and a stolen authorization code is worthless without the verifier.

Same destination, two roads: identical token endpoint, identical JWT claims (iss, sub, exp, scope) — you just took the insecure-but-simple route here so you could see the token from a single curl. Now open the simulator and watch the secure route reach the same place, and you'll understand both what a token is and why modern OAuth goes to such lengths to issue one safely.
