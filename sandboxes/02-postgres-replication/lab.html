Sandbox Lab·Module 3-4 — Databases

Postgres Indexing & Replication

Spin up a real Postgres primary + streaming read-replica with Docker, watch a sequential scan turn into an index scan, and prove that writes on the primary appear on the replica — the two moves that scale every read-heavy database.

Prerequisites

Docker Desktop must be running (check the whale icon / docker info succeeds). Then open a terminal and cd into this folder — every command below is run from sandboxes/02-postgres-replication/, right next to the docker-compose.yml.

01  What you'll build

Two Postgres containers wired into a streaming replication pair: a primary that takes writes and a read-replica that continuously replays the primary's write-ahead log (WAL). This is the exact pattern behind "add read replicas to scale reads" from the Module 3-4 notes.

There is no application code here — the whole topology is declared with a handful of Bitnami environment variables. You'll drive everything through docker compose exec … psql, so you're talking to real Postgres, not a mock.

          writes                      streaming WAL
  client ────────▶  ┌───────────────┐  ═══════════════▶  ┌───────────────┐
                    │   PRIMARY     │                     │   REPLICA     │
   reads ◀───────── │  (read/write) │                     │ (read-only)   │ ◀──── reads
                    │  service:     │                     │  service:     │
                    │   primary     │                     │   replica     │
                    │  :5432 -> host│                     │  :5432 -> 5433│
                    └───────────────┘                     └───────────────┘
                     bitnami/postgresql:16                bitnami/postgresql:16
                     REPLICATION_MODE=master              REPLICATION_MODE=slave
                                                          MASTER_HOST=primary

   • Writes (INSERT/UPDATE/DDL) go ONLY to the primary.
   • The primary ships its WAL to the replica over the Docker network.
   • The replica replays that WAL and serves read-only queries.
   • CREATE INDEX on the primary also replicates — the replica speeds up too.

02  The stack

Here is the complete docker-compose.yml — a modern Compose file (no obsolete top-level version: key), with pinned image tags. Replication is set up purely through environment variables; the replica runs pg_basebackup from the primary on first boot, then streams the WAL forever after.

# Postgres streaming replication sandbox — one PRIMARY, one READ-REPLICA.
# Replication is configured entirely through Bitnami env vars; NO custom code.

services:
  # PRIMARY — accepts writes and streams its WAL to every connected replica.
  primary:
    image: docker.io/bitnami/postgresql:16
    container_name: sd-pg-primary
    ports:
      - "5432:5432"                                   # optional host access for a GUI client
    environment:
      - POSTGRESQL_REPLICATION_MODE=master
      - POSTGRESQL_REPLICATION_USER=repl_user
      - POSTGRESQL_REPLICATION_PASSWORD=repl_password
      - POSTGRESQL_PASSWORD=postgres_password         # password for the `postgres` superuser
      - POSTGRESQL_DATABASE=labdb                     # a database to play in
    volumes:
      - primary_data:/bitnami/postgresql
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "postgres"]
      interval: 10s
      timeout: 5s
      retries: 12

  # REPLICA — read-only hot standby; pulls the WAL stream from `primary`.
  replica:
    image: docker.io/bitnami/postgresql:16
    container_name: sd-pg-replica
    depends_on:
      primary:
        condition: service_healthy
    ports:
      - "5433:5432"                                   # host 5433 -> replica's 5432
    environment:
      - POSTGRESQL_REPLICATION_MODE=slave
      - POSTGRESQL_REPLICATION_USER=repl_user
      - POSTGRESQL_REPLICATION_PASSWORD=repl_password
      - POSTGRESQL_MASTER_HOST=primary
      - POSTGRESQL_MASTER_PORT_NUMBER=5432
    volumes:
      - replica_data:/bitnami/postgresql

volumes:
  primary_data:
  replica_data:

Why the same replication creds on both sides

The replica authenticates to the primary as repl_user to open the WAL stream, so POSTGRESQL_REPLICATION_USER / POSTGRESQL_REPLICATION_PASSWORD must match on both services. POSTGRESQL_MASTER_HOST=primary resolves to the primary container by its Compose service name over the default network.

03  Bring it up

Start both containers in the background:

docker compose up -d

Confirm both are up and the primary is healthy:

docker compose ps

Watch the replica finish its base backup and switch to streaming — you're looking for a line like "database system is ready to accept read only connections":

docker compose logs replica | grep -i "read only"

The fastest proof it worked is to ask the primary who's connected as a replica — this should return one row whose state is streaming:

docker compose exec primary \
  psql -U postgres -d labdb -c "SELECT client_addr, state, sync_state FROM pg_stat_replication;"

If psql asks for a password

Local socket connections inside the container are trusted, so psql -U postgres normally just works. If you ever hit a prompt, the superuser password is postgres_password — or prefix the command with -e PGPASSWORD=postgres_password, e.g. docker compose exec -e PGPASSWORD=postgres_password primary psql -U postgres -d labdb.

04  The exercises

Four short exercises. Copy each block verbatim. Steps A, C and D run on the primary; step B runs on the replica to prove the data crossed over.

A
Create a table & bulk-insert 100k rows on PRIMARY

Open a shell into the primary and generate a realistic events table.

B
Read the same rows on REPLICA

Prove replication: the rows are already there, and writes are rejected.

C
Seq scan → index scan on PRIMARY

EXPLAIN ANALYZE before and after CREATE INDEX.

D
Measure replication lag on PRIMARY

Read pg_stat_replication to see how far behind the replica is.

Exercise A — create & bulk-insert (PRIMARY)

Drop into a psql session on the primary:

docker compose exec primary psql -U postgres -d labdb

Then paste this SQL. generate_series lets Postgres build all 100,000 rows server-side in one statement:

-- a simple analytics-style events table
CREATE TABLE events (
  id          bigserial     PRIMARY KEY,
  user_id     integer       NOT NULL,
  event_type  text          NOT NULL,
  payload     text,
  created_at  timestamptz   NOT NULL DEFAULT now()
);

-- bulk-insert 100,000 rows of pseudo-random traffic
INSERT INTO events (user_id, event_type, payload)
SELECT (random() * 10000)::int,
       (ARRAY['click','view','purchase','signup','logout'])[(random() * 4)::int + 1],
       md5(random()::text)
FROM generate_series(1, 100000);

SELECT count(*) FROM events;   -- expect: 100000

What to observe: the count is 100000. Leave this row count in mind — you'll check for the exact same number on the replica next. Type \q to exit (or keep it open in a second terminal tab).

Exercise B — read it back from the replica (REPLICA)

Now open a session on the replica and run read queries. You never inserted anything here — the rows arrived over the WAL stream:

docker compose exec replica psql -U postgres -d labdb
-- the data is already here, streamed from the primary
SELECT count(*) FROM events;                 -- expect: 100000
SELECT id, user_id, event_type FROM events ORDER BY id LIMIT 5;

-- confirm this really is a read-only standby:
INSERT INTO events (user_id, event_type) VALUES (1, 'nope');
-- ERROR:  cannot execute INSERT in a read-only transaction

What to observe: the count matches the primary's 100000 — replication works. The INSERT is rejected with "cannot execute INSERT in a read-only transaction", which is the whole point of a read-replica: it scales reads, and all writes must go to the primary. Exit with \q.

Exercise C — watch a seq scan become an index scan (PRIMARY)

Back on the primary. First, ask Postgres how it plans to find one user's events before any index exists:

docker compose exec primary psql -U postgres -d labdb
-- BEFORE the index: Postgres must scan every row
EXPLAIN ANALYZE
SELECT * FROM events WHERE user_id = 4242;

Look for Seq Scan on events and a large rows count / higher execution time — Postgres reads all 100k rows to find the handful that match. Now build an index on that column:

CREATE INDEX idx_events_user_id ON events (user_id);

Run the exact same query again:

-- AFTER the index: Postgres jumps straight to the matching rows
EXPLAIN ANALYZE
SELECT * FROM events WHERE user_id = 4242;

What to observe: the plan flips from Seq Scan to a Bitmap Index Scan / Index Scan using idx_events_user_id, and the actual time drops sharply — the query no longer touches every row. This is the single highest-leverage database tuning move: index the columns you filter and join on.

Bonus — the index replicates too

CREATE INDEX is DDL, and DDL travels over the WAL just like data. Re-run the EXPLAIN ANALYZE from Exercise B on the replica and you'll see it use idx_events_user_id as well — you tuned both databases with one statement.

Exercise D — measure replication lag (PRIMARY)

The primary tracks every attached replica in pg_stat_replication. Run this on the primary:

docker compose exec primary \
  psql -U postgres -d labdb -c "SELECT * FROM pg_stat_replication;"

For a readable lag view, compute the byte distance between what the primary has sent and what the replica has replayed:

docker compose exec primary psql -U postgres -d labdb -c "
SELECT client_addr,
       state,
       sync_state,
       pg_wal_lsn_diff(sent_lsn, replay_lsn) AS replay_lag_bytes
FROM pg_stat_replication;"

What to observe: one row, state = streaming, and replay_lag_bytes at or near 0 — the replica is caught up. To see lag appear, run the 100k-row INSERT from Exercise A again on the primary and immediately re-run this query: the byte gap briefly jumps, then settles back to ~0 as the replica catches up. That transient gap is replication lag — the reason read-replicas give you eventual, not strong, consistency.

05  Teardown

Stop and remove both containers and their data volumes so you start clean next time:

docker compose down -v

What to observe: Compose removes the containers, the default network, and the named volumes (primary_data, replica_data). Omit the -v if you want to keep the data and restart the same cluster later with docker compose up -d.

06  What this taught

  • Indexing turns an O(n) sequential scan into a targeted index lookup — you saw the query plan change and the time drop, live.
  • Streaming replication ships the primary's WAL to a read-only replica: writes go to one node, reads fan out across replicas. That's how you scale a read-heavy workload without sharding.
  • Replication lag is real and measurable in pg_stat_replication. A replica is eventually consistent, so reads that must be perfectly fresh have to hit the primary — the classic read-your-writes trade-off from the notes.
