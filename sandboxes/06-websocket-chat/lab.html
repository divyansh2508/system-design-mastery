Sandbox Lab·Module 10 — Real-Time

WebSocket Chat with Redis Fan-out

Run two identical chat servers behind an nginx load balancer and watch a Redis pub/sub channel fan every message out to clients on both instances — the pattern that lets real-time systems scale past a single box.

Prerequisites

Have Docker Desktop running (whale icon steady, not spinning). Then open a terminal and cd into this folder — every command below is run from sandboxes/06-websocket-chat/.

01  What you'll build

A tiny group chat that is horizontally scaled from the first minute: two copies of the same Node ws server, an nginx load balancer in front on host port 8080, and a Redis broker behind them carrying a single pub/sub channel.

When you open two browser tabs, nginx may route each to a different chat instance. A message typed in one tab has to reach a client whose socket lives on the other process entirely — a process it shares no memory with. Redis is the shared nervous system that makes that possible: every instance publishes what it receives and subscribes to everything, so each message fans out to all connected clients regardless of which server they landed on.

   Browser tab A                                    Browser tab B
        │   ws://localhost:8080/                           │
        └────────────────┬─────────────────┬───────────────┘
                         ▼                  ▼
                 ┌────────────────────────────────┐
                 │   nginx  :80   (host :8080)     │   round-robin LB
                 │   HTTP Upgrade → WebSocket      │   + WS upgrade headers
                 └────────┬───────────────┬────────┘
                          │               │
                 ┌────────▼─────┐  ┌───────▼──────┐
                 │   chat1      │  │   chat2      │    node:20-alpine + ws
                 │   :3000      │  │   :3000      │    (serves the HTML client)
                 └────────┬─────┘  └───────┬──────┘
                  PUBLISH │   SUBSCRIBE     │   (every instance does BOTH)
                          └───────┬─────────┘
                           ┌──────▼───────┐
                           │  redis :6379 │    pub/sub channel "chat"
                           └──────────────┘

The one idea

A WebSocket lives on one process. The moment you run more than one server, a client on instance A cannot hear a client on instance B — unless every instance relays messages through a shared broker. Redis pub/sub is that broker: publish what you receive, broadcast what you hear.

02  The stack

Four services, one file. Here is the docker-compose.yml that wires them together — modern Compose schema, no obsolete version: key, tags pinned, one comment per service:

# WebSocket Chat sandbox — two chat servers + Redis pub/sub + an nginx load balancer.
# Modern Compose schema (no top-level `version:` key — it is obsolete).

services:
  # Shared Redis broker: the pub/sub channel every chat instance fans messages through.
  redis:
    image: redis:7-alpine
    container_name: ws-chat-redis
    restart: unless-stopped

  # Chat instance #1 — a Node `ws` server that publishes to and subscribes from Redis.
  chat1:
    build: .
    image: ws-chat-lab:1.0
    container_name: ws-chat-1
    environment:
      - INSTANCE_ID=chat1
      - REDIS_URL=redis://redis:6379
    depends_on:
      - redis
    restart: unless-stopped

  # Chat instance #2 — the identical image run as a second replica, so messages must cross Redis.
  chat2:
    build: .
    image: ws-chat-lab:1.0
    container_name: ws-chat-2
    environment:
      - INSTANCE_ID=chat2
      - REDIS_URL=redis://redis:6379
    depends_on:
      - redis
    restart: unless-stopped

  # nginx load balancer: round-robins WebSocket connections across chat1/chat2, published on host :8080.
  nginx:
    image: nginx:alpine
    container_name: ws-chat-nginx
    ports:
      - "8080:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - chat1
      - chat2
    restart: unless-stopped

The rest of the folder is the small custom image these two chat containers build from, plus the load-balancer config:

FileRole
DockerfileBuilds the node:20-alpine image and installs deps
package.jsonDeps: ws (WebSockets) + redis (pub/sub client)
server.jsThe chat server: ws sockets, Redis publish/subscribe, and it serves its own HTML client
nginx.confUpstream of chat1+chat2 with the WebSocket Upgrade headers, listen 80

Why the Upgrade headers matter

A WebSocket starts as an ordinary HTTP/1.1 request carrying Upgrade: websocket. For the proxy to let that protocol switch through, nginx.conf sets proxy_http_version 1.1, proxy_set_header Upgrade $http_upgrade, and proxy_set_header Connection "upgrade". Drop those and the socket silently fails to connect.

03  Bring it up

Build the image and start all four containers in the background:

docker compose up -d --build

The first run downloads redis:7-alpine, nginx:alpine, and node:20-alpine and installs the npm deps, so give it a minute. When it returns, confirm every container is up:

docker compose ps

You want to see four services — redis, chat1, chat2, nginx — all Up, with nginx publishing 0.0.0.0:8080->80/tcp. Peek at the chat logs to confirm both instances connected to Redis and are listening:

docker compose logs chat1 chat2

Each should print a line like:

[chat1] chat server listening on http://0.0.0.0:3000
[chat2] chat server listening on http://0.0.0.0:3000

Port 8080 already in use?

If something else owns port 8080, change the left side of the nginx mapping (e.g. "9090:80") in docker-compose.yml, re-run docker compose up -d, and use that port everywhere below.

04  The exercises

The whole point is to see a message cross the process boundary. Run these in order, keeping the browser tabs and a log window on screen at the same time.

1 · OpenTwo tabs, maybe two instances

Open the chat client in two separate browser tabs, both pointed at the load balancer:

# macOS — run twice to get two tabs
open http://localhost:8080
open http://localhost:8080

# Linux: xdg-open http://localhost:8080   ·   Windows: start http://localhost:8080
# …or just paste http://localhost:8080 into two tabs by hand.

What to observeEach tab's header reads "you are connected to chat1" or "…chat2". Because nginx round-robins new connections, the two tabs often land on different instances — that's exactly the split you want to test. (If both show the same one, open a third tab; they alternate.)

2 · ChatA message crosses the instance boundary

In the tab connected to chat1, type something and hit Send. Then send one from the chat2 tab too.

# Nothing to type in the terminal here — use the two browser tabs.
# tab A (chat1):  "hello from tab A"
# tab B (chat2):  "and hi from tab B"

What to observeBoth messages appear in both tabs — even though the two clients are held by two different server processes. Each message is tagged "handled by chat1" or "handled by chat2", naming the instance that first received it and published it to Redis. That tag is your proof the message travelled through the broker.

3 · TraceWatch the fan-out in the logs

Now follow the logs live and send a few more messages, watching which instance does what:

docker compose logs -f

What to observeFor a single message you'll see it handled once and delivered everywhere. The instance that owns the sending tab prints the publish; both instances print the fan-out they got back from Redis:

chat1  | [chat1] recv from client -> publish: hello from tab A
chat1  | [chat1] fan-out from redis <- chat1: hello from tab A
chat2  | [chat2] fan-out from redis <- chat1: hello from tab A

One publish, two fan-out lines — one per instance. chat2 delivered a message it never received from a browser; it only heard it on the Redis channel. Press Ctrl-C to stop following.

4 · ProveWhy Redis is not optional at scale

Convince yourself the broker is load-bearing by cutting it out from under the running stack:

docker compose stop redis

Send a message from either tab, then bring Redis back:

docker compose start redis

What to observeWith Redis stopped, messages stop crossing — a chat server has no other way to tell its peer what happened, and the two instances go deaf to each other. Restart Redis and the fan-out resumes. This is the lesson in one move: the instant you run more than one WebSocket server, in-memory broadcast is not enough — you need a shared pub/sub layer (Redis here; Kafka, NATS, or a managed bus in production) to reunite clients scattered across processes.

05  Teardown

Stop and remove all four containers, the network, and any named volumes in one shot:

docker compose down -v

Run docker compose ps afterward — it should list nothing for this project. The built image lingers in your local cache; drop it too with docker image rm ws-chat-lab:1.0 if you want a spotless machine.

06  What this taught + where next

You now have felt, not just read, the core scaling problem of real-time systems:

  • WebSockets are stateful and process-local. A connection lives inside one server's memory; a plain in-process broadcast only ever reaches that server's own clients.
  • A load balancer scatters clients. nginx round-robins connections across instances, so any two users are likely on different servers — and that's the whole reason a single-process broadcast breaks.
  • Pub/sub reunites them. Every instance publishes what it receives and subscribes to everything, so Redis fans each message out to all instances, which then broadcast to their own sockets. Publish once, deliver everywhere.
  • The upgrade dance is real. Fronting WebSockets with a proxy means forwarding the HTTP/1.1 Upgrade/Connection headers — a detail that bites everyone once.

In a system design interview, this is exactly the muscle you flex on "how do you scale a chat / notifications / live-feed service past one server?" — the answer is a stateless-ish socket tier plus a shared pub/sub backbone, and now you've watched it work.
